# ADR-0001: メール検証機能を自前実装する

## ステータス

承認済み（2026-03-14）

## コンテキスト

Issue #40 でメール検証機能（アカウント登録時の確認メール送信 → リンククリックで有効化）を実装するにあたり、以下の選択肢を検討した。

- 自前実装（crypto/rand トークン生成 + SMTP 送信）
- Go 認証ライブラリへの委譲（Authboss 等）
- 外部認証サービスの利用（Clerk, Auth0, Supabase Auth 等）
- 自前ホストの ID プロバイダー導入（Ory Kratos, Keycloak 等）

### 評価した選択肢

| 選択肢 | メール検証 | 導入コスト | 懸念 |
|---|---|---|---|
| **自前実装** | ○ | 低 | セキュリティの自己責任 |
| Authboss | ○（confirm モジュール） | 中 | HTTP ハンドラ層を乗っ取る設計で Clean Architecture と衝突。開発者1名のプロジェクト |
| Clerk | ○（フルマネージド） | 低 | SaaS 依存。フロントの auth UI が大幅に変わる。10k MAU 超で課金 |
| Auth0 | ○（フルマネージド） | 低〜中 | $0.07/MAU で個人プロジェクトにはコスト高 |
| Supabase Auth | ○ | 中 | Go SDK が pre-release で本番非推奨 |
| Ory Kratos | ○（リンク/コード両対応） | 高 | Docker サービス追加。YAML 設定が膨大。現段階では過剰 |
| Keycloak | ○ | 非常に高 | Java ベースでリソース消費大。個人プロジェクトには不向き |

## 決定

**自前実装を採用する。**

## 理由

1. **既存アーキテクチャとの整合性**: Clean Architecture + DDD のレイヤー構成（domain/repository インターフェース → infrastructure 実装）に自然に組み込める。外部ライブラリは HTTP ハンドラ層を乗っ取るものが多く、既存の設計と衝突する。

2. **導入コストと複雑性**: 外部サービス（Clerk, Auth0）は SaaS 依存が生じ、自前ホスト（Kratos, Keycloak）は Docker サービス追加と設定の複雑性が高い。現段階の要件（メール検証のみ）に対して過剰。

3. **実装の安全性**: 自前実装でも以下のセキュリティ要件を満たしている。
   - `crypto/rand` による暗号学的に安全なトークン生成（32バイト / hex 64文字）
   - トークンの有効期限管理（24時間）
   - DB インデックスによるトークン検索
   - メール列挙攻撃を防ぐ再送エンドポイント設計（常に 200 を返す）

4. **プロジェクト規模**: 個人開発プロジェクトであり、当面は大規模なユーザーベースを想定しない。MFA・OAuth・Passkey 等の高度な認証機能は現時点では不要。

## 今後の対応

### 短期（自前実装の強化）

- [ ] 再送エンドポイントへのレート制限追加
- [ ] 本番環境向けメール配信サービスの選定（SendGrid, Resend 等）

### 長期（移行の判断基準）

以下のいずれかの条件を満たした場合、外部サービスへの移行を再検討する。

- MFA（多要素認証）や Passkey 対応が必要になった場合
- OAuth プロバイダー（Google, GitHub 等）によるソーシャルログインが必要になった場合
- ユーザー数の増加によりセキュリティ監査が必要になった場合

移行先の第一候補:
- **Clerk**: DX が良く、Go SDK が安定しており、フルマネージドで運用負荷が低い
- **Ory Kratos**: 自前ホストで完全な制御を維持したい場合。ヘッドレス設計が Clean Architecture と相性が良い
